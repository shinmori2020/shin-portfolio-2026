# Works スクリーンショット自動撮影（設計メモ）

Works 掲載サイトのスクリーンショットを Playwright で統一条件撮影する仕組みの設計メモ。運用手順は `scripts/README.md` を参照し、ここには**なぜその実装なのか**と**踏んだ落とし穴**を残す。

関連ファイル：

| ファイル | 役割 |
|---|---|
| `scripts/capture-list.mjs` | 撮影対象の案件リスト（人間が実URLを記入する設定側） |
| `scripts/capture-works.mjs` | 撮影本体（統一条件を固定。案件ごとの事情は list 側で吸収する） |
| `src/lib/workImages.ts` | `public/works/{slug}/cover.*` `full.*` を探索する解決層 |
| `src/components/common/WorkImageLightbox.tsx` | full があればライトボックス表示・無ければ導線ごと非表示 |

## 2種類の画像と役割分担

| 出力 | 撮り方 | 用途 |
|---|---|---|
| `cover.png` | ファーストビュー（`fullPage: false`） | 一覧カード・詳細ヒーロー・最新作カード |
| `full.png` | ページ全体（`fullPage: true`） | 詳細ページの「全体を見る」ライトボックス |

`capture-list.mjs` は slug ごとに2件（cover / full）を並べる。出力名は `out`（既定 `cover.png`）で切り替える。1ターゲット＝1スクリーンショットの単純な構造を保ち、両方を1回で撮る特殊化はしていない。

## 落とし穴1: fullPage 撮影で reveal 前の白紙が写る

### 症状

`fullPage: true` で撮ると、ファーストビューから下の広い範囲が**白紙**で写った（headless-wp-media = nordic-works）。見出しだけ出て、記事カード・特集カード・導入事例カードの中身が一切写らない。

### 原因

対象サイトの fade-in が **Framer Motion の `whileInView` を `once: true` 無しで実装**していた。この場合、要素はビューポートに入ると表示され、**出ると `opacity: 0` に戻される**。

`fullPage` はビューポート外までまとめて撮るだけで、スクロールしたことにはならない。よって画面外の要素は一度も発火せず `opacity: 0` のまま写る。

さらに厄介なのは、**撮影前に全体をスクロールしても解決しない**こと。スクロールで発火はするが、先頭へ戻した時点で下部が再びビューポート外になり、隠れ直してしまう。

診断で、先頭復帰後に残っていた9要素はすべて `opacity: 0; transform: translateY(16px)` のインラインスタイルを持ち、正体は記事/特集/導入事例のカードそのものだった。これが `whileInView` の署名である。

### 対策（2段構え）

`fullPage` の時だけ、撮影直前に以下を行う（`cover` は表示済みなので何もしない）。

1. **全体をゆっくりスクロールする** … 遅延読み込み（`loading=lazy`・クライアント取得）を誘発し、一度きりの reveal を発火させる
2. **隠れ直した reveal を最終状態へ確定する** … 「インラインで opacity がほぼ0 **かつ** transform 付き」という `whileInView` の署名を持つ要素だけを対象に、インラインスタイルを上書きする

条件を「opacity と transform の両方」に絞るのが要点。意図的に隠している要素（`display: none`、transform を持たないオーバーレイ等）を巻き込まないための線引きである。

### 不採用にした案：ビューポートを全高へ広げる

「ビューポート自体をページ全高にすれば全要素が入りっぱなしになり、隠れない」という案を実際に試したが**不採用**。

- `100vh` 指定のヒーローがビューポート高（5580px）まで**引き伸ばされ**、実際の見た目と別物になった
- しかも白紙は直らなかった
- 加えて Chromium のテクスチャ上限（約 16384 device px / DSF=2 なら CSS px で約 8192）に当たるリスクがある

ページの見た目を保ったまま撮る、という目的に反するため破棄した。

## 落とし穴2: この環境の npm は `-- <flag>` をスクリプトへ渡さない

`npm run capture -- --skip-existing` を実行しても、スクリプトの `process.argv` にフラグが**入らない**（`--list` でも同様に再現）。結果、既存を残すつもりが**全件再撮影**され、承認済みの `cover.png` を上書きしてしまった。

**フラグ付き実行は node を直接叩くこと。**

```bash
node scripts/capture-works.mjs --skip-existing
node scripts/capture-works.mjs --slug=headless-wp-media
```

フラグ無しの `npm run capture`（全件撮影）は問題なく動く。

なお `--skip-existing` の存在判定は cover 固定ではなく**ターゲットの `out` 単位**で行う。cover が既にあり full が無い時、cover はスキップして full だけ撮る、という運用ができる。

## Cookie バナーの扱い

`dismiss` に閉じるボタンのセレクタを書くと、撮影前にクリックして閉じる。

- 4案件を調査した結果、**バナーが出るのは nordic-works のみ**。他3案件には `dismiss` を書いていない（存在しないセレクタを書くと毎回「見つからず」の警告が出るだけで無意味なため）
- 閉じ方は **「拒否する」** を採用。撮影目的では見た目に差が無く、不要な同意を残さないため
- `cover` / `full` の**両方**に適用している。片方だけ消すと、一覧サムネにはバナーがあるのに全体表示では無い、という食い違いが起きる

full では特に重要で、バナーは固定配置のためページ中腹に焼き込まれ、「最新の記事」の見出しに重なっていた。

## ライトボックスのフォールバック

`WorkImageLightbox` は `full` がある時だけクリック可能にし、「全体を見る」のヒントとモーダルを出す。**`full` が無い案件では導線ごと出さない**（静的なブラウザ枠のみ）。

cover へフォールバックして「全体を見る」を出すと、開いてもファーストビューが出るだけで期待を裏切るため、導線自体を消す判断とした。画像規約（`public/works/README.md`）の「置くだけで反映・無ければ自動フォールバック」の思想に沿っている。

## WebP 化（対応済み）

全要素が描画されるようになった分 `full.png` は重くなり、特に headless-wp-media は約9.9MB あった。ライトボックスを開いた時に実際に読み込まれるため、表示速度を優先する本サイトの方針に反する。

`scripts/to-webp.mjs` で cover / full の8枚すべてを WebP へ変換した。画像規約（`src/lib/workImages.ts`）は `webp` を最優先で探索するため、**置き換えるだけでコード変更は不要**。

```bash
node scripts/to-webp.mjs              # 変換のみ（PNG を残して画質を確認したい時）
node scripts/to-webp.mjs --delete-png # 変換後に元 PNG を削除
```

### 実績（quality=80 / effort=6）

| 画像 | 変換前 | 変換後 | 削減 |
|---|---|---|---|
| headless-wp-media/full | 9,626 KB | 970 KB | -89.9% |
| headless-wp-media/cover | 1,578 KB | 83 KB | -94.8% |
| multilingual-ec/full | 1,346 KB | 326 KB | -75.8% |
| multilingual-ec/cover | 718 KB | 32 KB | -95.5% |
| **合計（8枚）** | **14,685 KB** | **1,868 KB** | **-87.3%** |

削減率が極端に高いものは、UI のベタ塗り・グラデーションに対して PNG が非効率だったため。等倍で目視確認し、細線のロゴ・日本語の文字・写真のディテールいずれも劣化は認められなかった。

### 運用メモ

- **PNG は残さない**。`webp` が優先探索されるため PNG は永遠に未使用となり、リポジトリを膨らませるだけになる
- 撮影スクリプトの出力は PNG のまま（撮影と変換は別工程）。**再撮影したら変換もやり直す**
- 画質を疑う時は `--delete-png` を付けずに実行し、PNG と並べて比較してから削除する
