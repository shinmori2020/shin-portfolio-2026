# Contact フォーム状態フィードバック実装 — 統合指示書

このフォルダには Contact ページ用の3コンポーネント(完成コード)と統合例が入っている。
**コードは設計済み・レビュー済み。ゼロから書き直さず このコードをベースに既存プロジェクトの規約へ合わせる調整のみ行うこと。**

## 含まれるファイル

| ファイル | 内容 |
|---|---|
| `components/SubmitButton.tsx` | 送信ボタンの状態遷移(idle / loading / success / error)。スピナーとチェック描画付き |
| `components/TextField.tsx` | インラインバリデーション付きテキスト入力。エラー文のフェード表示 |
| `components/ConsentCheckbox.tsx` | チェックマークがストローク描画される同意チェックボックス |
| `components/ContactForm.example.tsx` | 3つを統合したフォーム実装例(バリデーションロジック込み) |

## 設計ルール(変更しないこと)

1. **モーションの基準値**: 180ms / easeOut。サイト全体のページ遷移フェードと同じ速度に揃えている
2. **CLSゼロ**: ボタンは `min-w` + `h` 固定。エラー文の領域は `min-h` で常時確保。レイアウトを動かす変更は不可
3. **reduced-motion 等価**: `useReducedMotion()` で分岐済み。動きを消しても情報が欠けない設計(スピナー→「送信中…」テキスト、チェック→描画済みで即表示)。この分岐を削除しないこと
4. **a11y**: `aria-live` / `aria-invalid` / `aria-describedby` / 送信時の最初のエラー欄へのフォーカス移動。すべて維持すること
5. **バリデーションのタイミング**: blurで初回検証 → エラー後は入力のたび再検証(reward early, punish late)。この非対称性は意図的
6. シェイク演出・ホバーでの浮き上がり(translateY)・影は禁止(プロジェクト制約)

## 統合時にやること

1. 既存の Contact ページ実装(フォームのマークアップと送信処理)を先に読んで現状を把握する
2. `components/` 配下の3ファイルをプロジェクトのコンポーネント置き場(例: `src/components/form/`)へ移動し importパスをプロジェクトのエイリアス(`@/` など)に合わせる
3. 色の置き換え: コード内の `green-950` / `green-900` / `stone-*` / `red-800` は仮置き。プロジェクトの実際のブランドトークン(深緑・オフホワイト・罫線色)へ置換する。`design-reference/Home.dc.html` の `:root` トークンを正とする
4. `ContactForm.example.tsx` を参考に既存の Contact ページへ組み込む。既存のフィールド(会社名 / ご依頼の種類 / ご予算 / 希望納期 / お問い合わせ内容)は既存のまま活かし、TextField のバリデーションパターンを必須項目(お名前 / メール / 内容 / 同意)へ展開する
5. 送信先 `/api/contact` は既存の実装(またはフォームサービス)に合わせて差し替える
6. 既存のボタンコンポーネントと二重定義にならないよう確認。既存ボタンに矢印スライドのホバーがあるなら idle 状態の ArrowIcon と統合してよい

## 完了条件

- `tsc --noEmit` と lint がエラーなしで通る
- 送信フロー4状態(idle → loading → success / error → idle)が動作する
- 空送信で全エラーが表示され 最初のエラー欄にフォーカスが移る
- DevTools の Rendering > Emulate CSS media feature `prefers-reduced-motion: reduce` で動きが消えても状態が読み取れる
- エラー表示の前後でレイアウトシフトが起きない(ボタン寸法・エラー領域の高さが固定されている)
